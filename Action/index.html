Kinstrife Blueprints & Mechanics

"com.kinstrife.core.mechanics": "file:/GameDev/Core/Packages/Mechanics" The purpose of this package is to provide a system that handles most boilerplate and authoring needs for (for now linear) state-based behaviours, sequenced events and (semi-)complex interactions.

Intro

If building a game's mechanics is building various cars for various functions, then using this package is to build cars of the same brand, from the same blueprint and stock components. Whether it's a 4WD, an automatic or manual, you will still use a combustion engine, a steering wheel and brakes.

Much in the same way, we can boil down a game's mechanics, large and small, into its base components, all using a shared dataset and universal 'language' set expressed into standardized components, systems and authoring & development work flow.

In Unity terminology, it combines the Animator with a bit of Timeline and a lite visual scripting solution.

This package consists of two parts - Blueprints (authoring) and Mechanics (runtime).

Blueprints provides an extensible data framework and visual authoring workflow to define the data and their relationships as expressed through the games mechanics.

Mechanics converts the blueprints into runtime entities from a conversion step. Although it uses ECS at its core and is fully multi threaded, its full functionality is exposed for managed components through an event broadcaster.

In order to keep Mechanics general-purpose and applicable to as many in game mechanics as possible, game mechanics are boiled down into the following standardized principles:

Context (Owner, root) Mechanic (action, behavior, state) Input/Output Resource Provider/Resource Signaling (external) It is highly encouraged to build custom logic by hooking into mechanics' many callbacks in custom systems/components, using helper systems and extending its blueprint dataset where relevant.
