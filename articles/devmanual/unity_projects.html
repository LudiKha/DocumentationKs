Managing Unity projects

Â 

Rather than having a single (huge) Unity project for all of our development, our package-based setup allows us to have various projects with shared code and assets. Unity's editor slows down considerably with larger projects and in many cases, compile times are also longer - as a result, it is often preferable to develop in smaller (partial) projects.

With our setup, projects by themselves do not contain any assets or scripts. Additionally, most project settings are symlinked to our Core project, located at Core\Projects\KinstrifeCore. The only relevant information in projects is the list of packages in their project manifest (located in the project's Packages folder). Projects of course have their library folder, which stores imported assets - as a result, it makes sense to have multiple projects around with readily imported assets, rather than using a single project while constantly swapping (and re-importing) packages.

In short, Unity projects should generally be seen as workspaces rather than an all-encompassing sources of truth as in most other Unity development projects.

Opening projects

Projects can be opened via the Unity hub. All missing folders (which are ignored by git) will be automatically generated.

Creating projects

Create a new project as you regularly would via the Unity hub. After doing so, run the project setup wizard located at Core\Projects\KinstrifeCore\SetupUnityProject.bat, providing the path of the newly created package as input when prompted.

Some settings will be replaced by symlinks to the KinstrifeCore project settings. This is the case e.g. with the TagManager.asset, to ensure that all projects use the same physics layer names and setup. The list of project settings to link is defined in SetupSettingsToSymlink.txt in the Core folder.
